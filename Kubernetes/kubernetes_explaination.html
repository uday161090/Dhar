01

What is Kubernetes?

Definition

The Automated Ops Team

Kubernetes is a tool that automatically runs, monitors, and manages your apps inside containers — so you don't have to do it by hand. It handles crashes, traffic spikes, deployments, and failures 24/7.

🍕 PizzaFast Analogy You run PizzaFast — an online pizza ordering site. On Super Bowl Sunday, 50,000 orders flood in. Without Kubernetes, you'd be manually spinning up servers at midnight. With Kubernetes, it detects the surge, automatically launches 40 more copies of your app, and removes them when traffic dies down. You sleep. 😴

❌ Without Kubernetes

Manual server setup · Crash at 3 AM means downtime · Scaling requires SSH · Deploy = "site is down" · Passwords in .env files · No self-healing

✅ With Kubernetes

Auto-restart on crash · Scale up in seconds · Zero-downtime rolling deploys · Secrets managed safely · Health checks built-in · Always self-healing

Containers — Portable App Boxes

Foundation

Container = App + Everything It Needs

A container packages your app code, runtime, libraries, and settings into one portable box. It runs identically on your laptop, a test server, or a cloud machine.

📦 Moving House Analogy You pack your kitchen items into one clearly labelled box — pots, knives, recipe books, all together. At the new house, open the box and the kitchen works immediately. A container is that box, but for your app.
📌 Example — Dockerfile for PizzaFast
# Recipe for building the PizzaFast container
FROM node:18                    # start from Node.js 18
WORKDIR /app                   # working folder inside container
COPY package*.json .           # copy dependency list
RUN npm install                # install dependencies
COPY . .                       # copy all app files
CMD ["node", "app.js"]        # start command

# Build & run:
# docker build -t pizzafast:v1 .
# docker run -p 3000:3000 pizzafast:v1
FeatureContainerVirtual Machine
SizeMegabytesGigabytes
Start timeMilliseconds–Seconds1–5 minutes
What's insideApp + libraries onlyFull OS + App
PortabilityRuns the same anywhereCan vary by environment
DensityHundreds per machine10–20 per machine

Control Plane — The Brain

Control Plane

API Server — The Front Door

Every request — from kubectl, dashboards, CI/CD pipelines — enters through the API Server. It authenticates you, checks permissions, validates your YAML, then saves it to etcd.

🏥 Hospital Reception Analogy Every patient (request) must check in at reception first. The receptionist verifies your ID (auth), checks your appointment (RBAC), and routes you to the right doctor (component). No one enters the operating theatre without checking in.
📌 Example — kubectl apply flow
# You type:
kubectl apply -f pizzafast-deployment.yaml

# API Server does, in order:
1. Receives HTTPS POST /apis/apps/v1/deployments
2. Checks your certificate → WHO ARE YOU?
3. Checks RBAC → ARE YOU ALLOWED?
4. Validates the YAML schema → IS THIS VALID?
5. Writes to etcd → SAVED.
6. Returns: "deployment.apps/pizzafast created"
Control Plane

etcd — The Cluster's Memory

etcd is a distributed key-value database that stores everything Kubernetes knows — how many apps are running, their config, which nodes exist, all secrets. If etcd dies without a backup, your cluster has no memory.

🧠 Long-Term Memory Analogy etcd is Kubernetes' long-term memory. Every goal, every config, every secret is stored here. Lose etcd → Kubernetes forgets everything. That's why you always back it up.
📌 What etcd stores for PizzaFast
/registry/deployments/default/pizzafast
  replicas: 5
  image: pizzafast:v3
  status: 5/5 running

/registry/secrets/default/db-password
  password: <base64 encoded>

/registry/nodes/worker-node-1
  cpu_free: 3.2 cores
  memory_free: 6.4 GB
  status: Ready
⚠️ Always Back Up etcd in Production! ETCDCTL_API=3 etcdctl snapshot save backup.db
Managed K8s (EKS, GKE, AKS) backs this up for you automatically.
Control Plane

Scheduler — The Smart Matchmaker

When a new Pod needs to run, the Scheduler finds the best Worker Node for it based on available CPU, memory, placement rules, and affinities.

📌 Scheduler picking a Node for PizzaFast
# New pizzafront Pod needs: 500m CPU, 256MB RAM

# Scheduler scores each node:
Worker Node 1: 200m CPU free  → TOO SMALL, skip
Worker Node 2: 800m CPU free  → FITS! Best balance.
Worker Node 3: 600m CPU free  → fits, but node 2 is better

# Scheduler writes to etcd:
pizzafront-pod-xyz → assigned to Worker Node 2

# kubelet on Node 2 picks this up instantly
# and starts pulling the container image
Control Plane

Controller Manager — The Watchdog

Runs endless loops watching the cluster. Its job: make sure what is running always matches what you asked for. Finds gaps → fills them. This is Kubernetes' self-healing superpower.

🐕 Night Security Guard Analogy The guard checks: "there should always be 5 lights on in corridor B". If one is off, they replace the bulb immediately. They never stop doing rounds. They never take a break.
📌 Controller Manager fixing PizzaFast at 3 AM
# You set: replicas: 5 in your Deployment
# 3:17 AM — Worker Node 2 runs out of memory
# Pod pizzafast-xk9q2 is killed

# Controller Manager loop (runs every ~5s):
desired replicas: 5
currently running: 4  ← GAP DETECTED!

# Action taken automatically:
→ Creates new Pod object in etcd
→ Scheduler assigns to Node 1
→ kubelet pulls image and starts container
→ 3:17:28 AM — Back to 5 running. 11 seconds total.
# You find out in the morning from the logs.

Worker Nodes — Where Apps Live

Worker Node

kubelet — The Node's Floor Manager

An agent running on every Worker Node. Watches the API Server for assignments, tells the container runtime to start/stop containers, checks health probes, and reports status back.

📌 kubelet starting a PizzaFast container
1. Scheduler assigns pod → Worker Node 2
2. kubelet notices via API Server watch
3. kubelet reads Pod spec:
     image: pizzafast:v3
     cpu: 500m, memory: 256Mi
4. Tells containerd: "pull pizzafast:v3"
5. containerd downloads & starts container
6. kubelet runs readiness probe: GET /health
7. Gets 200 OK → reports "Pod is Ready"
8. kube-proxy adds Pod to Service LB
9. Real customer traffic starts hitting this Pod
Worker Node

kube-proxy — Traffic Router

Manages iptables rules on every Node. Routes incoming Service traffic to the correct Pod. Instantly updates rules when Pods are added or removed.

📌 kube-proxy routing PizzaFast orders
# order-service has 3 Pods:
Pod A: 10.244.1.10
Pod B: 10.244.2.15
Pod C: 10.244.3.8

# kube-proxy iptables rules:
33% → Pod A
33% → Pod B
33% → Pod C

# Pod B crashes → rules update:
50% → Pod A
50% → Pod C  ← instant update
Worker Node

Pod — Smallest Deployable Unit

Core Concept

Pod = Container Wrapper

A Pod wraps one or more containers that share the same network and storage. Most Pods contain just one container — your app. Pods are temporary — they come and go; never depend on a specific Pod being alive.

🚢 Shipping Container Analogy A Pod is a shipping container on a cargo ship. Inside is your cargo (app container). The ship (Worker Node) carries many shipping containers. The port authority (Control Plane) decides which ship each container goes on. You don't book a specific ship — just the container size.
📌 Pod for PizzaFast's web frontend
apiVersion: v1
kind: Pod
metadata:
  name: pizzafront-pod
  labels:
    app: pizzafront    # ← label used by Services to find this Pod
    tier: frontend
spec:
  containers:
  - name: web
    image: pizzafast-frontend:v3
    ports:
    - containerPort: 3000
    resources:
      requests:
        cpu: "200m"      # minimum: 0.2 of a CPU core
        memory: "128Mi"  # minimum: 128 MB RAM
      limits:
        cpu: "500m"      # maximum: 0.5 of a CPU core
        memory: "256Mi"  # maximum: 256 MB RAM
⚠️ Never Create Bare Pods in Production Bare Pods have no owner. If the Node dies, the Pod is gone forever — nothing recreates it. Always use a Deployment (next section) which manages Pods automatically.

Deployment — Run Apps Reliably

Workload

Deployment = Desired State Contract

You declare "I always want 5 copies of my app running". Kubernetes keeps that true — forever. Rolling updates and instant rollbacks built in.

🛵 Staffing Agency Analogy A Deployment is a staffing agency contract: "I always need 5 pizza delivery drivers on shift." If a driver quits, the agency finds a replacement immediately. To upgrade to electric bikes (new version), they swap them out one-by-one so delivery never stops. If the bikes break, you call: "go back to regular bikes" — that's a rollback.
📌 Deployment for PizzaFast order service
apiVersion: apps/v1
kind: Deployment
metadata:
  name: order-service
spec:
  replicas: 5              # always keep 5 Pods running
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1    # at most 1 Pod down during update
      maxSurge: 1           # at most 1 extra Pod during update
  selector:
    matchLabels:
      app: order-service
  template:
    spec:
      containers:
      - name: order-service
        image: pizzafast-orders:v4
        ports:
        - containerPort: 8080
📌 Deployment commands for PizzaFast
# Deploy or update from YAML
kubectl apply -f order-service.yaml

# Watch the rolling update happen live
kubectl rollout status deployment/order-service

# Scale to 20 copies for Super Bowl Sunday
kubectl scale deployment/order-service --replicas=20

# Push new version (triggers rolling update automatically)
kubectl set image deployment/order-service \
  order-service=pizzafast-orders:v5

# Oh no — v5 has a bug. Instant rollback to v4:
kubectl rollout undo deployment/order-service

Service — Stable Networking

Networking

Service = Permanent Phone Number

Pods get new IPs every restart. A Service gives a stable DNS name and IP to a group of Pods. Even as Pods come and go, the Service address never changes.

📞 Hospital Phone Number Analogy The hospital's main number (Service) stays the same for years. Individual doctors (Pods) retire, take leave, move. When you call, reception routes you to whoever is available right now. You never dial a doctor's personal number.
📌 Four Service types for PizzaFast
# 1. ClusterIP — internal only (most common)
# order-service → database-service (inside cluster)
type: ClusterIP
# DNS: database-service.default.svc.cluster.local

# 2. LoadBalancer — public internet access
# pizzafast.com → frontend Pods
type: LoadBalancer
# Gets a real public IP from AWS/GCP/Azure

# 3. NodePort — dev/testing only
type: NodePort
nodePort: 30080
# Access at http://node-ip:30080

# 4. Ingress — smart HTTP router
# One LB → routes by URL path/hostname
# pizzafast.com/     → frontend-service
# pizzafast.com/api  → api-service
# pizzafast.com/admin → admin-service

Storage — Persistent Data

Storage

PVC — Your Storage Request

Containers lose all data when they stop (like a whiteboard). A PersistentVolumeClaim requests storage that survives container restarts — essential for databases.

🗄️ Filing Cabinet Analogy Containers = whiteboards (wiped clean each time). PersistentVolume = filing cabinet that survives office renovations. PVC = your request: "I need a cabinet with 4 drawers (100GB)." Kubernetes finds one and locks it to your Pod.
📌 Persistent storage for PizzaFast PostgreSQL
# Step 1: Request 100GB storage
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: postgres-data
spec:
  accessModes: [ReadWriteOnce]
  resources:
    requests:
      storage: 100Gi

# Step 2: Mount it in the Pod
volumeMounts:
- mountPath: /var/lib/postgresql/data
  name: db-storage
volumes:
- name: db-storage
  persistentVolumeClaim:
    claimName: postgres-data
# Data survives container crashes and node moves!

Auto-Scaling — Handle Any Traffic

HPA

Horizontal Pod Autoscaler

The HPA watches CPU/memory and automatically adds or removes Pods. You set min, max, and target — Kubernetes handles the rest.

📞 Call Centre Analogy Target: each agent handles ≤70% capacity. 9 AM Monday: 3 agents. Noon Friday: 20 agents. 2 AM Saturday: 2 agents. No manager makes these calls — it's fully automatic.
📌 HPA timeline on Super Bowl Sunday
5:00 PM  3 Pods running, CPU 15%
6:30 PM  Game starts — orders flood in 🏈
6:31 PM  CPU hits 80% → HPA adds 2 Pods
6:32 PM  CPU still 75% → HPA adds 3 more → 8 Pods
6:35 PM  CPU settles at 58% → HPA holds at 8
10:30 PM Orders slow → CPU 20% → HPA removes 5
12:00 AM Back to 3 Pods. Normal night. 💤

# HPA config:
minReplicas: 3
maxReplicas: 50
targetCPU: 60%

Health Checks — Know When You're Sick

Probes

Three Types of Health Checks

ProbeQuestionFails → Action
LivenessIs it still alive?Restart container
ReadinessReady for traffic?Remove from LB
StartupFinished starting?Keep waiting
📌 PizzaFast real scenario
# DB connection pool exhausted:
/ready returns 503  → readiness fails
→ Pod removed from Service (no customer errors)
/health returns 200  → Pod is alive, just busy
→ No restart. Just waiting.

# When DB recovers:
/ready returns 200  → Pod rejoins Service
→ Traffic flows again. Zero manual intervention.

RBAC — Who Can Do What

Security

Role-Based Access Control

Controls who (users, services) can do what (create, delete, view) on which resources (Pods, Secrets, Deployments) in which namespace.

🔑 Keycard System Analogy Role = what doors the keycard opens. RoleBinding = assigning a keycard to a person. Intern: kitchen + bathroom only. Engineer: office floors + server room. Admin: everything. Define keycards once, hand them out.
📌 RBAC for PizzaFast team
# Role: junior devs can only VIEW in 'development'
kind: Role
rules:
- resources: ["pods", "services"]
  verbs: ["get", "list", "watch"] # read-only

# Assign to Alice:
kind: RoleBinding
subjects:
- name: [email protected]

# Result:
kubectl get pods -n development     ✓ works
kubectl delete pod xyz -n development ✗ FORBIDDEN
kubectl get pods -n production      ✗ FORBIDDEN

Helm — App Store for K8s

Package Manager

Install Complex Apps in One Command

Helm packages up all the YAML files needed for an app into a Chart. Install something like PostgreSQL (normally 10+ YAMLs) with a single command.

📱 App Store Analogy Without Helm: manually download source, compile, configure each component, wire them together — for every app. With Helm: tap "Install PostgreSQL" → done in 30 seconds.
📌 Installing monitoring stack for PizzaFast
# Add chart repository
helm repo add prometheus-community \
  https://prometheus-community.github.io/helm-charts

# Install Prometheus + Grafana + AlertManager
# (would take 15+ YAMLs manually)
helm install pizzafast-monitoring \
  prometheus-community/kube-prometheus-stack \
  --set grafana.adminPassword="myPass" \
  --namespace monitoring

# Upgrade to newer chart version:
helm upgrade pizzafast-monitoring \
  prometheus-community/kube-prometheus-stack

# Complete teardown of everything:
helm uninstall pizzafast-monitoring -n monitoring
